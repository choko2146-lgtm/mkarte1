# Codex Workflow

## 作業終了時の必須更新

今後の開発では、作業終了ごとに以下のMarkdownを確認し、必要があれば必ず更新する。

- `docs/current_status.md`
- `docs/error_notes.md`
- `docs/next_tasks.md`

更新方針:

- `current_status.md`: 完了済み機能、作業途中機能、現在のStep、実機確認状況を更新する。
- `error_notes.md`: Android Studio、Gradle、Room migration、Emulator、Git、実機不具合などを蓄積する。
- `next_tasks.md`: 次に実装するStep、保留中の課題、レビュー待ち事項を更新する。

Markdown更新後に変更がある場合は、ユーザーへcommitするか確認する。pushはユーザー確認後に行う。

## 基本ルール

- 調査のみを依頼された場合、コード変更は禁止。
- 実装はStepごとに進める。
- 各Step完了後は必ず報告し、レビュー待ちで止まる。
- レビュー前に次のStepへ進まない。
- 実装前に既存コード、既存仕様、既存導線を確認する。
- 変更理由を報告に含める。
- エラー発生時は原因、対処、再発防止を記録する。
- 不明点があり、推測で進めると危険な場合は質問する。

## 実装時の進め方

1. 対象ファイルを確認する。
2. 既存の命名、Repository、DAO、Activityの流れに合わせる。
3. 最小差分で実装する。
4. 指定外のファイルを触らない。
5. `assembleDebug` を実行する。
6. 変更ファイル、変更内容、確認結果を報告する。

## 禁止事項

- DB構造を無断で変更しない。
- Entityを無断で変更しない。
- DBバージョンを無断で上げない。
- 写真保存方式を無断で変更しない。
- 外部ストレージ権限をむやみに追加しない。
- 既存機能に関係ない大規模リファクタリングをしない。
- ユーザー未確認のまま次Stepへ進まない。

## エラー対応

エラーが出た場合は、以下を必ず整理する。

- エラー内容
- 発生条件
- 原因
- 修正内容
- 修正後の確認結果
- 影響範囲

重要なエラーは `docs/error_notes.md` に追記する。

## Git運用

- 重要な仕様、作業状況、エラー対処はMarkdownで残す。
- 作業単位が完了したらcommitする。
- pushはユーザー確認後に行う。
- 既存の未確認変更を勝手に戻さない。

## 報告フォーマット

Step完了時は以下を含める。

1. 変更したファイル一覧
2. 実装内容
3. 既存機能への影響
4. `assembleDebug` の結果
5. 実機確認ポイント
6. 次Stepへ進む前の注意点
